## 0. 공통 데이터 스키마

모든 모듈이 주고받는 데이터의 형태를 먼저 확정한다.
이 스키마가 전체 파이프라인의 계약(contract)이다.

### 0.1 법령 객체 (Law)

```json
{
  "law_id": "act_20260101_12345",
  "name": "금융소비자보호법",
  "short_name": "금소법",
  "type": "법률",
  "law_category": "금융법",
  "enacted_date": "2020-03-24",
  "last_amended_date": "2025-01-01",
  "effective_date": "2025-07-01",
  "total_articles": 73,
  "source": "legalize-kr",
  "source_version": "2026-05-16",
  "structure": {
    "편": [],
    "장": [
      { "number": 1, "title": "총칙", "article_range": [1, 5] },
      { "number": 2, "title": "금융상품판매업등", "article_range": [6, 11] }
    ]
  },
  "articles": [ /* Article 객체 배열 */ ]
}
```

#### law_category 판정 규칙

자동 판정 후 수동 오버라이드 가능.

```
규칙 1: 법령명 키워드 매칭 (우선순위순)
  "금융|은행|보험|투자|신용|증권|자본시장|여신|수신" → 금융법
  "공공기관|정부기관|공기업|공단|진흥" → 공공기관법
  "소송|절차|재판|심판|조정|중재" → 절차법
  "민법|상법|계약|채권|물권|담보|보증" → 민사법
  매칭 없음 → 일반

규칙 2: 키워드 복수 매칭 시
  "금융" + "소송" → 금융법 (금융이 우선)
  "공공기관" + "계약" → 공공기관법 (기관이 우선)

규칙 3: 수동 오버라이드
  config/law_category_override.json에 {law_id: category} 등록
  예: { "act_국가계약법": "공공기관법" }
```

## 1. 입력 모듈 (①)

### 1.1 입력 경로 상세

#### 경로 A: 기수집 법령 선택 (Phase 1 주력)

```
사용자 → 법령명 또는 법령ID 입력
         ↓
[검색] legalize-kr DB에서 매칭
  ├─ 정확 매칭: 즉시 로드
  ├─ 유사 매칭: 후보 목록 제시 (예: "금소" → "금융소비자보호법" 제안)
  └─ 미매칭: "법령을 찾을 수 없습니다. 법제처에서 검색하시겠습니까?" → 경로 B
         ↓
[로드] 법령 전문 텍스트 로드
         ↓
[변환] Law 객체 생성
         ↓
② 조문 파서로 전달
```

**검색 로직:**
```
1차: 정확 매칭 (법령명 == 입력값)
2차: 포함 매칭 (법령명.contains(입력값))
3차: 약칭 매칭 (short_name 테이블 조회)
     약칭 테이블 예: { "금소법": "금융소비자보호법", "상법": "상법", "국계법": "국가계약법" }
4차: 초성 매칭 (ㄱㅅㅂ → 금소법) — Phase 2
```

**약칭 테이블 구축:**
Phase 0에서 수집한 16개 법령 + 법제처 공식 약칭 목록에서 초기 약칭 테이블 생성.
legalize-kr 1,680건 전체에 대해서는 Phase 1 중 점진적 확장.

#### 경로 B: 법제처 실시간 수집 (Phase 2, 설계만)

```
사용자 → 법령명 입력
         ↓
[API] 법제처 법령정보센터 API 호출
  - 엔드포인트: http://www.law.go.kr/DRF/lawSearch.do
  - 파라미터: target=law, query={법령명}, type=JSON
         ↓
[응답] 법령 XML/JSON 수신
         ↓
[파싱] 법제처 포맷 → Law 객체 변환
  - 법제처 XML 구조: <법령><조문><조문단위> 계층
  - 조항호목이 이미 태그로 구분되어 있으므로 파싱 부담 적음
         ↓
② 조문 파서로 전달 (파서의 "사전파싱" 모드)
```

**법제처 API 제약사항:**
- 일 호출 한도: 미공식이나 과도한 호출 시 차단
- 응답 속도: 2~5초/건
- 데이터 포맷: XML 기본, JSON 옵션

#### 경로 C: 직접 입력 (Phase 2, 설계만)

```
사용자 → 텍스트 붙여넣기 또는 파일 업로드
         ↓
[검증] 입력 검증
  - 최소 길이: 100자 이상
  - 최대 길이: 500,000자 (상법 수준)
  - 조문 구조 존재 여부: "제N조" 패턴 최소 1건 탐지
         ↓
[메타데이터] 사용자 입력
  - 법령명 (필수)
  - 법령 유형 (선택, 미입력 시 자동 판정)
  - 시행일/개정일 (선택)
         ↓
② 조문 파서로 전달 (파서의 "원문파싱" 모드)
```

**내부규정 대응 특이사항:**
내부규정은 법령과 조문 번호 체계가 다를 수 있다.
- "제N조" 대신 "제N항" 또는 "N." 만 사용하는 경우
- 편/장 구조 없이 평문 나열인 경우
- "부칙" 대신 "경과조치" 또는 "시행일" 단순 기재
→ Phase 2에서 "내부규정 모드" 파서 별도 개발 필요

### 1.2 입력 유효성 검증

모든 경로에서 공통 적용하는 검증 규칙:

| 검증 항목 | 규칙 | 실패 시 처리 |
|----------|------|------------|
| 텍스트 비어있음 | len(text) > 0 | 에러: "법령 텍스트가 비어있습니다" |
| 조문 구조 부재 | regex `제\d+조` 매칭 ≥ 1 | 경고: "조문 구조를 인식할 수 없습니다. 내부규정 형식인지 확인하세요" |
| 인코딩 | UTF-8 확인 | 자동 변환 시도 → 실패 시 에러 |
| 중복 분석 | 동일 law_id + 동일 source_version | 경고: "이미 분석된 법령입니다. 재분석하시겠습니까?" |

## 2. 조문 파서 (②)

### 2.1 파서의 역할

입력 텍스트를 Law 객체의 articles 배열로 변환한다.
핵심은 조/항/호/목 계층을 정확하게 분리하는 것이다.

### 2.2 파싱 모드

| 모드 | 입력 | 특징 |
|------|------|------|
| pre-parsed | legalize-kr DB의 이미 구조화된 데이터 | 조문 경계 확정됨. 항/호 분리만 수행 |
| raw-legal | 법제처 API 또는 직접 입력 텍스트 | 전체 파싱 수행 |
| pre-tagged | 법제처 XML (태그로 구조 표시) | 태그 기반 변환, 파싱 불필요 |

### 2.3 파싱 규칙 (raw-legal 모드)

한국 법령 텍스트의 구조적 특징에 기반한 파싱 규칙이다.

#### Rule P-01: 조(Article) 분리

```
패턴: /^제(\d+)조(의(\d+))?\s*[\(（]([^)）]+)[\)）]/gm
매칭: "제10조(설명의무)" → number_raw=10, insert=null, title="설명의무"
매칭: "제10조의2(추가 의무)" → number_raw=10, insert=2, title="추가 의무"

분리 전략:
  텍스트를 "제N조" 패턴 위치에서 split
  각 split 결과가 하나의 Article
  마지막 Article은 다음 "제N조" 또는 "부칙" 또는 EOF까지
```

**엣지 케이스:**
```
Case 1: 제목 없는 조문
  "제5조 이 법에서 사용하는 용어의 뜻은..."
  → title: null, full_text 첫 문장에서 추론

Case 2: 삽입조 깊이
  "제10조의2의3" → insert_depth = 2
  "제10조의23" → insert_depth = 1 (의23이 하나의 숫자)
  판별: "의" 다음에 오는 숫자를 분리
  정규식: /제(\d+)((?:의\d+)*)/
  "제10조의2의3" → group2 = "의2의3" → split("의") → depth=2

Case 3: 조문 내 타법 인용에 "제N조" 등장
  "「국가재정법」 제40조에 따라"
  → 「」 괄호 내부의 "제N조"는 조문 분리 대상이 아님
  처리: 「」 내부를 먼저 마스킹 → 조문 분리 → 마스킹 해제
```

#### Rule P-02: 항(Paragraph) 분리

```
패턴: /^([①②③④⑤⑥⑦⑧⑨⑩⑪⑫⑬⑭⑮⑯⑰⑱⑲⑳])/gm
매칭: "① 금융상품판매업자등은..." → para number="①"

분리 전략:
  조문 텍스트를 ① ② ③ 기호 위치에서 split
  기호가 없는 조문 = 항이 1개 (전문이 ①항)
```

**엣지 케이스:**
```
Case 1: 원문자 21 이상
  법령에 따라 ㉑㉒㉓... 사용 (드물지만 존재)
  대비: 원문자 패턴을 ①~⑳ + ㉑~㉟ + 21~99(아라비아숫자 항) 모두 포함

Case 2: 번호 없는 단일 항
  "제5조 이 법에서..." (항 번호 없이 바로 본문)
  → paragraphs = [{ number: null, text: full_text }]

Case 3: 본문 중 원문자 인용
  "제①항의 규정에 불구하고" ← 본문 내 참조
  → 행 시작 위치(^)에서만 매칭하여 구분
```

#### Rule P-03: 호(Item) 분리

```
패턴: /^\s*(\d+)\.\s+/gm
매칭: "  1. 금융상품의 내용" → item number="1"

분리 전략:
  각 항 텍스트를 "N." 패턴 위치에서 split
  들여쓰기(2칸 이상) + 숫자 + 마침표 조합을 호로 인식
```

**엣지 케이스:**
```
Case 1: 본문 중 "1." 등장 (날짜, 조문번호)
  "2025. 1. 1.부터 시행한다" ← 날짜 표기
  "제10조 제1항 제1호" ← 참조 표기
  → 행 시작 + 들여쓰기 위치에서만 매칭
  → 날짜 패턴 /\d{4}\.\s*\d{1,2}\.\s*\d{1,2}\./ 선 제외

Case 2: 호가 다행에 걸치는 경우
  "  1. 금융상품의 내용과
       그 위험에 관한 사항"
  → 다음 호("2.") 시작 전까지를 하나의 호로 통합

Case 3: 가목나목 (Sub-item)
  "    가. 예금성 상품"
  → /^\s*(가|나|다|라|마|바|사|아|자|차|카|타|파|하)\.\s+/
  → item의 sub_items 배열에 추가
```

#### Rule P-04: 부칙(Addendum) 처리

```
패턴: /^부\s*칙/gm

처리:
  부칙 시작 이후 텍스트는 articles 배열에 포함하지 않음
  별도 addendum 필드에 저장
  부칙은 패턴 스캔 대상에서 제외 (부칙은 경과규정이므로 개선 대상 아님)
```

#### Rule P-05: 편/장/절 구조 추출

```
패턴:
  편: /^제(\d+)편\s+(.+)/gm
  장: /^제(\d+)장\s+(.+)/gm
  절: /^제(\d+)절\s+(.+)/gm

처리:
  Law 객체의 structure 필드에 저장
  각 조문의 chapter 필드에 소속 장/절 기록
  스캔에는 직접 사용하지 않으나, 리포트에서 위치 참조용
```

### 2.4 파서 출력 검증

파싱 완료 후 아래 검증을 수행한다.

| 검증 | 규칙 | 실패 시 |
|------|------|--------|
| 조문 수 일관성 | 파싱된 조문 수 ≥ 1 | 에러: 파싱 실패 |
| 조문 번호 연속성 | 빈 번호 5개 이상 연속 = 파싱 오류 가능 | 경고 로그 |
| 빈 조문 | full_text 길이 < 10자 | 경고: 파싱 오류 가능 |
| 호 카운팅 교차검증 | 항 내 호 개수 vs 정규식 매칭 수 일치 | 불일치 시 재파싱 |

### 2.5 파서 성능 요구사항

| 중규모 (금소법) | ~73 | < 0.3초 |

**S-02. 위임 검증 (S-02+L-04 통합)**

```
설계 원칙:

  누락 없이 규정되어 있으면 문제가 아니다.
  문제는 (A) 하위법령 미이행, (B) 위임 범위 불명확(포괄위임)일 때만 발생한다.

로직 (3단계):
  ──────────────────────────────────────────────────
  1단계: 위임 식별 (데이터 수집 — Finding 미생성)
  ──────────────────────────────────────────────────
    각 조문에서 primary 키워드 매칭

  ──────────────────────────────────────────────────

    (기존 L-04 로직을 여기서 실행 — L-04는 독립 패턴에서 S-02 하위 단계로 통합)

  ──────────────────────────────────────────────────
  3단계: 통합 판정
  ──────────────────────────────────────────────────
    Case A: 위임 전건 이행 + 포괄위임 없음 → 양호 (Finding 미생성)
    Case B: 하위법령 미제정 → 심각/경고 (위임미이행)

등급 판정:
  미이행 3건↑ → 심각
  미이행 1~2건 → 경고
  부분이행 → 경고
  포괄위임만 → 주의
  전건 이행 → 양호 (위임 비율 무관)

핵심 변경점:
  ✗ 기존: 위임 "비율"이 높으면 문제 → 폐기
  ✓ 변경: 위임 "이행 여부"가 핵심. 비율은 판단 기준이 아님
  ✓ L-04(위임미이행)는 독립 패턴에서 S-02의 하위 단계로 흡수
```

**S-03. 모호 표현**

**F-02. 면책 조항**

```
대상: 각 조문
키워드 셋:
  패턴A: "책임을 지지 아니한다|책임이 없다|책임을 면한다|면책"
  패턴B: "손해를 배상하지 아니한다|배상.*책임.*없"
  패턴C: "일체의|모든|어떠한" + (패턴A 또는 패턴B)

로직:
  1. 패턴C 매칭 → 전면 면책 (고의·중과실 포함 가능)
  2. 패턴A/B 매칭 → 부분 면책
  3. 동일 조문에 "고의 또는 중과실" "고의·중과실" 예외 표현 존재 여부 확인
     → 있으면 정상, 없으면 오탐 아닌 진짜 면책

등급 판정:
  패턴C + 고의·중과실 예외 없음 → 심각
  패턴A/B + 고의·중과실 예외 없음 → 경고
  패턴A/B + 예외 있으나 범위 불명확 → 주의
  고의·중과실 예외 있음 → 양호
```

```
  약한 처분: ["시정권고", "경고", "주의", "개선명령"]

로직:
  1. 키워드 매칭 + 강도 분류
  2. 사전 청문/이의절차 규정 존재 여부 확인
     → 동일 조문 또는 해당 법령 내 "청문" "의견제출" "이의신청" 조문 존재 여부
  3. 처분 기준(별표/별도 기준) 존재 여부 확인

등급 판정:
  강한 처분 + 청문 절차 없음 → 심각
  강한 처분 + 청문 있으나 기준 불명확 → 경고
  중간 처분 + 절차 미비 → 주의
  약한 처분 → 개선
  절차 완비 → 양호
```

```

  3. 동일 조문 내 참조 수 카운팅

등급 판정:
  1개 조문에 서로 다른 법률 5건↑ 참조 → 경고 (독해 곤란)
  1개 조문에 서로 다른 법률 3~4건 → 주의
```

```
    5. 모니터링: "자체점검|모니터링|사후관리"

  매칭된 요소 수 카운팅

등급 판정:
  5대 요소 중 0개 매칭 → 심각 (내부통제 부재)
  1~2개 매칭 → 경고
  3개 매칭 → 주의
  4개 매칭 → 개선
  5개 매칭 → 양호
```

**G-05. 보고 의무**

```
키워드 셋: ["보고", "제출", "통보", "공시", "신고", "통지", "공개"]

로직:
  1. 의무 조항("하여야 한다") 중 보고 키워드 포함 조문 식별
  2. 보고 주기 규정 여부: "매년|분기|반기|월|수시" 탐지
  3. 보고 양식 규정 여부: "별지|서식|양식" 탐지
  4. 제출처 규정 여부: "~에게|~에 제출" 탐지

등급 판정:
  의무 있으나 보고 규정 전무 → 심각
  주기/양식/제출처 중 2개↑ 누락 → 경고
  1개 누락 → 주의
  양식만 미비 → 개선
```

#### 효율성 카테고리 (4패턴 — 룰 only)

**E-01. 조건 중첩**

```
대상: 각 조문
탐지 대상: 복합 조건 구조

키워드 셋:
  조건 접속: ["및", "또는", "이고", "이며", "하고", "하며"]
  조건 도입: ["경우", "때", "경우에는", "때에는", "요건"]
  중첩 표지: ["에 해당하는 경우로서", "충족하고", "갖추어야 하며",
              "다음 각 호의", "모두 충족"]

로직:
  1. 조건 접속사 수 카운팅
  2. "다음 각 호의 요건을 모두 충족" 패턴 → 호 수가 조건 수
  3. 중첩 깊이: 조건문 안에 다시 조건문 (재귀 구조)

등급 판정:
  조건 5단계↑ 중첩 → 심각
  조건 3~4단계 → 경고
  AND/OR 관계 불명확 → 주의
  조건 2단계 → 개선
```

**E-02. 서식 관련**

```
키워드 셋: ["별지 제", "서식", "별표", "양식"]

등급 판정:
  서식 20종↑ (법령 전체) → 심각
  서식 변경에 시행규칙 개정 필요 → 경고
  비전자화 서식 → 주의
  불필요 기재 항목 → 개선
```

**E-03. 아날로그 잔재**

```
키워드 셋:
  강한 아날로그: ["서면으로", "날인", "인감", "대면하여", "직접 출석"]
  중간 아날로그: ["우편", "등기우편", "내용증명"]
  약한 아날로그: ["서면", "문서"]  ← 전자문서 포함 가능 표현이 없는 경우만

로직:
  1. 키워드 매칭
  2. 동일 조문에 전자적 대안 허용 표현 존재 여부
     탐지: "전자문서|전자적 방법|정보통신망|전자서명|온라인"
     → 대안 있으면 등급 하향

등급 판정:
  강한 아날로그 + 전자 대안 없음 → 심각
  강한 아날로그 + 전자 대안 있으나 동일 효력 미규정 → 경고
  중간 아날로그 + 대안 없음 → 주의
  약한 아날로그 → 개선
```

**E-04. 유형별 규제 차등**

```
대상: 법령 전체 단위 분석
로직:
  1. 법령 유형 키워드 사전 로드 (config/type_keywords.json)
  2. 법령 텍스트에서 유형 키워드 셋 추출
  3. 유형 키워드가 동일 조문에 2개↑ 공존하면 → 차등 규제 조문
  4. 법령 전체에서 차등 규제 조문 비율 산출

유형 키워드 사전 예시:
  공공기관운영법: ["공기업", "준정부기관", "기타공공기관", "시장형", "준시장형",
                   "위탁집행형", "기금관리형"]
  부동산투자회사법: ["자기관리", "위탁관리", "기업구조조정", "개발전문"]
  → 법령별로 사전 등록 필요. Phase 0 스캔 결과에서 추출 가능.

등급 판정:
  유형 7개↑ + 거의 모든 조문 관통 → 심각
  유형 5~6개 → 경고
  유형 3~4개 → 주의
  유형 2개 → 개선

  미등록 법령 (사전에 없는 법령) → 스킵 (Finding 미생성)
```

### 3.3 오탐 보정 레이어 (False Positive Correction)

Phase 0에서 발견된 오탐 원인을 구조적으로 해소한다.

#### FPC-01: 인용법령 화이트리스트

```
문제: 「」 매칭이 법률명 인용과 용어정의를 구분하지 못함
해법: 화이트리스트 매칭

화이트리스트 소스:
  1. legalize-kr 법령 목록 1,680건의 법령명
  2. 주요 법률 약칭 (금소법, 자본시장법 등)

보정 규칙:
  「X」 매칭 시 X가 화이트리스트에 있으면 → 정상 인용 (L-01)
  X가 화이트리스트에 없으면 → 용어정의 또는 강조 부호 가능 → 제외

예외: 「X」 뒤에 "제N조" 가 오면 → 화이트리스트 무관하게 타법참조(L-02)로 분류
```

#### FPC-02: 용어정의 조문 제외

```
문제: 용어정의 조문(통상 제2조)에서 다수 키워드 오탐
해법: 용어정의 조문 식별 후 일부 패턴 적용 제외

용어정의 조문 판별:
  제2조 + 제목에 "정의|용어" 포함
  또는 본문에 "이 법에서 사용하는 용어의 뜻" 포함

제외 패턴:
  L-01 (인용법령): 용어정의 내 법령 인용은 정상적 정의 행위
  S-03 (모호표현): 정의 조문의 "등" "기타"는 열거의 일부
  
제외하지 않는 패턴:
  S-04 (열거과다): 용어정의라도 호가 30개 이상이면 구조 문제
  G-01 (예외단서): 정의 조문의 예외는 적용범위 혼란 초래
```

#### FPC-03: 절차법 제재공백 필터

```
문제: 절차법(민소/형소/행소)은 자체 벌칙이 적은 것이 정상
해법: 법령 유형이 "절차법"이면 E-05(제재공백) 패턴 등급을 2단계 하향

적용:
  law_category == "절차법" 일 때:
    E-05 심각 → 주의
    E-05 경고 → 개선
    E-05 주의 → 양호 (제외)
    
  추가 필터: 해당 의무가 "소송절차상 의무" (기일출석, 서면제출 등)이면
    → 절차 하자로 규율되므로 제재공백 아님 → 양호
```

#### FPC-04: 벌칙 조문의 키워드 오탐

```
문제: 벌칙 조문(통상 마지막 장) 자체에서 "금지" "제한" 키워드가 잡힘
  예: "다음 각 호의 어느 하나에 해당하는 자는 3년 이하의 징역에 처한다"
  → 벌칙 조문은 제재 규정이지 권리 제한이 아님

해법: 벌칙 조문 식별 후 F-01(권리제한), F-03(처분명령) 제외

벌칙 조문 판별:
  장 제목에 "벌칙|과태료|벌금|징역" 포함
  또는 조문 내 "징역|벌금|과태료에 처한다" 포함
```

#### FPC-05: 메타규정 위임 보정

```
문제: 약관규제법, 행정기본법 등 "다른 법을 규율하는 법"은 위임이 원래 적음
  → 위임 비율이 낮은 것이 정상이며, 오히려 과도 위임이 문제

해법: 영향 없음 (위임과다 패턴은 위임이 많을 때 플래그하므로 메타규정은 자연 양호)
  → 별도 보정 불필요. 기록 목적으로만 문서화.
```

## 4. LLM 진단 엔진 (④)

### 4.1 역할 범위

LLM은 두 가지 역할을 수행한다:

```
역할 A: 정밀 판단 (3패턴)
  - F-04 의사표시의제: 룰이 "간주/본다" 후보 추출 → LLM이 진짜 의제인지 판단
  - F-05 자의적재량: 룰이 "할 수 있다" 후보 추출 → LLM이 자의적 재량인지 판단
  - E-05 제재공백: 룰이 의무↔벌칙 불일치 추출 → LLM이 진짜 공백인지 판단

역할 B: 맞춤 권고안 생성 (등급 심각·경고 전체)
  - 핵심 설계서 §2.5에서 설계 완료
```

### 4.2 역할 A: 정밀 판단 프롬프트

#### F-04. 의사표시의제 판단

```
[시스템 프롬프트]
당신은 한국 약관·소비자보호 법제 전문가입니다.
주어진 조문이 "의사표시 의제" (침묵을 동의로 간주하는 규정)에 해당하는지 판단합니다.

[판단 기준]
1. "~한 것으로 본다" "~한 것으로 간주한다" 표현이 있더라도,
   아래의 경우는 의사표시 의제가 아닙니다:
   - 법적 지위 의제: "공무원으로 본다" (신분 의제)
   - 사실 추정: "과실이 있는 것으로 추정한다" (증명책임 전환)
   - 산정 기준: "취득가액을 영(零)으로 본다" (계산 편의)

2. 의사표시 의제에 해당하는 경우:
   - "동의한 것으로 본다" (묵시적 동의)
   - "승낙한 것으로 본다" (부작위에 의한 계약 성립)
   - "이의가 없는 것으로 본다" (이의 의제)
   - "갱신된 것으로 본다" (자동 갱신)

3. 의제가 맞더라도, 아래 완화 요소가 있으면 등급 하향:
   - 사전 통지 의무 규정 존재
   - 의제 전 합리적 기간(30일↑) 부여
   - 철회 가능 기간 명시

[입력]
법령명: {law_name}
조문 전문: {article_full_text}
룰 매칭 키워드: {matched_text}

[응답 형식 — JSON만, 다른 텍스트 없이]
{
  "is_deemed_consent": true/false,
  "reasoning": "판단 근거 (2문장)",
  "deemed_type": "동의의제|승낙의제|이의부재의제|갱신의제|해당없음",
  "mitigating_factors": ["사전통지 규정 존재", ...],
  "severity": "심각|경고|주의|개선|양호",
  "severity_basis": "등급 판정 근거 (1문장)"
}
```

#### F-05. 자의적 재량 판단

```
[시스템 프롬프트]
당신은 한국 행정법 전문가입니다.
주어진 조문의 재량 규정이 "자의적 재량"에 해당하는지 판단합니다.

[판단 기준]
1. "~할 수 있다"가 있더라도, 아래는 자의적 재량이 아닙니다:
   - 기속행위: 요건이 구체적으로 열거되고, 효과도 특정 ("~ 경우에는 ~할 수 있다")
   - 허용 규정: 사인(私人)에게 자유를 부여 ("당사자는 ~할 수 있다")
   - 재량 기준 명시: "다음 각 호를 고려하여" + 고려 요소 열거

2. 자의적 재량에 해당하는 경우:
   - 발동 요건 없이 포괄적 재량: "필요하다고 인정하는 경우"
   - 행정청 재량 + 기준 부재: "장관은 ~할 수 있다" (요건 없음)
   - 재량 범위 무제한: "적절한 조치를 할 수 있다"

3. 등급 판정 시 고려 요소:
   - 재량 행사 결과의 중대성 (기본권 관련 > 행정 편의)
   - 이유 부기 의무 유무
   - 사후 통제 장치 (보고, 공시, 이의신청) 유무

[입력]
법령명: {law_name}
조문 전문: {article_full_text}
룰 매칭 키워드: {matched_text}

[응답 형식 — JSON만]
{
  "is_arbitrary_discretion": true/false,
  "reasoning": "판단 근거 (2문장)",
  "discretion_type": "기속재량|자유재량|포괄재량|해당없음",
  "subject": "행정청|사인|기관",
  "impact_level": "기본권|재산권|행정편의",
  "control_mechanisms": ["이유부기", "보고의무", ...],
  "severity": "심각|경고|주의|개선|양호",
  "severity_basis": "등급 판정 근거 (1문장)"
}
```

#### E-05. 제재공백 판단

```
[시스템 프롬프트]
당신은 한국 법령체계 전문가입니다.
주어진 의무 조항에 대응하는 제재 규정이 존재하는지 판단합니다.

[판단 기준]
1. 아래의 경우는 제재공백이 아닙니다:
   - 타법 벌칙 적용: "이 법 위반 행위는 형법 제X조를 적용한다"
   - 절차법의 절차적 제재: 의무 위반 → 절차 하자 → 재판 효력 부정
   - 행정상 불이익: 위반 → 인허가 거부/취소 등 간접 제재
   - 선언적 의무: 법적 강제력 없는 훈시 규정 ("노력하여야 한다")

2. 제재공백에 해당하는 경우:
   - "하여야 한다" 의무 + 동일 법령 내 벌칙/과태료 조항 부재
   - 벌칙 조항은 있으나 해당 의무가 벌칙 대상 목록에서 누락

3. "노력하여야 한다" vs "하여야 한다" 구분:
   - "노력" 포함 → 훈시 규정 → 제재공백 아님
   - "하여야 한다" → 법적 의무 → 제재 대응 확인 필요

[입력]
법령명: {law_name}
의무 조문: {obligation_article_text}
해당 법령의 벌칙/과태료 조문 목록: {penalty_articles_summary}

[응답 형식 — JSON만]
{
  "has_sanction_gap": true/false,
  "reasoning": "판단 근거 (2문장)",
```

### 4.3 배치 처리 아키텍처

```
[Finding 목록 (LLM 대상)]
        ↓
   ┌────────────┐
   │ Batcher    │  ── 동일 패턴 건을 묶어 배치 구성
   │            │     배치 크기: 5건/회 (프롬프트 토큰 상한 고려)
   └────────────┘
        ↓
   ┌────────────┐
   │ Rate       │  ── API 호출 속도 제어
   │ Limiter    │     Sonnet: 4,000 RPM 상한 → 동시 호출 3건 제한
   │            │     Gemini Flash: 별도 제한
   └────────────┘
        ↓
   ┌────────────┐
   │ API Caller │  ── 실제 API 호출
   │            │     타임아웃: 30초/건
   │            │     재시도: 최대 2회 (JSON 파싱 실패 시)
   └────────────┘
        ↓
   ┌────────────┐
   │ Response   │  ── 응답 검증 + 파싱
   │ Validator  │
   └────────────┘
        ↓
   [Finding 업데이트 (llm_adjusted 필드)]
```

#### 배치 구성 규칙

```
원칙: 동일 패턴 건을 모아서 호출 (시스템 프롬프트 재사용)

배치 1: F-04 건 전체 (의사표시의제)
  → 시스템 프롬프트 1번 + 조문 N건 순차 처리

배치 2: F-05 건 전체 (자의적재량)

배치 3: E-05 건 전체 (제재공백)
  → E-05는 벌칙 조문 목록을 공통 컨텍스트로 1회만 포함

배치 4~N: 역할 B (맞춤 권고안 생성)
  → 카테고리별로 묶어 처리
```

#### 토큰 예산 관리

```
법령 1건 기준 예산 상한: 150K 입력 토큰

배분:
  역할 A (정밀 판단): ~50K (최대 25건 × 2K)
  역할 B (맞춤 권고안): ~80K (최대 40건 × 2K)
  여유: ~20K

초과 시 처리:
  1. 역할 B 대상을 등급 심각만으로 축소
  2. 그래도 초과 시 Gemini Flash로 전환
  3. 최종 fallback: Layer 1 템플릿만 사용
```

### 5.2 Tool 1: check_article_exists 구현

```
입력:
  law_name: str     — 인용 법령명 (「」 내부 텍스트)
  article_num: str  — 인용 조문 번호 ("10" 또는 "10의2")

처리 흐름:
  1. law_index에서 법령명 매칭
     - 정확 매칭 → 확인
     - 약칭 매칭 → 확인
     - 유사 매칭 (edit distance ≤ 3) → 후보 제시

  2. 매칭된 법령의 article_numbers에서 조문 존재 확인

  3. 법령 자체가 없는 경우:
     - 폐지 법령 목록(별도 관리) 대조
     - 제명 변경 목록 대조
     - 모두 미매칭 → "확인 불가" (법제처 API fallback)

출력:
  {
    "exists": true/false/null,
    "status": "exists|not_found|law_repealed|law_renamed|unknown",
    "current_law_name": "현행 법령명 (변경된 경우)",
    "note": "참고 사항"
  }
```

### 5.3 Tool 2: check_enforcement_decree 구현

**※ 설계 변경**: L-04(위임미이행)는 독립 패턴에서 S-02(위임검증)의 하위 단계로 통합됨.
이 Tool은 S-02의 2단계(하위법령 이행 검증)에서 호출된다.

```
입력:
  law_name: str          — 법률명
  article_num: str       — 위임 조문 번호
  delegated_content: str — 위임 내용 ("대통령령으로 정하는 바에 따라" 등)

처리 흐름:
  1. law_index에서 has_enforcement_decree 확인
     → false이면 → 시행령 자체 미제정 → "none"

  2. true이면 enforcement_decree_name으로 시행령 인덱스 조회
     → 시행령의 article_numbers에서 관련 조문 탐색
     → 법률 제N조 → 시행령 제N조 근방 (±3조) 탐색 (조문번호 대응 관례)

  3. 시행령 조문이 있으면 → "full" 또는 "partial"
     판별 기준: 시행령 조문의 조문 수 / 법률 위임 건수 비율
     → 비율 80%↑ = full, 50~80% = partial, < 50% = partial

출력:
  {
    "decree_exists": true/false,
    "decree_name": "시행령명",
    "coverage": "full|partial|none",
    "matched_decree_articles": ["10","11","12"],
    "note": "비고"
  }
```

### 5.4 법제처 API Fallback


## 6. 에러 처리 전략

### 6.1 에러 분류

| 등급 | 정의 | 예시 | 처리 |
|------|------|------|------|
| Fatal | 분석 진행 불가 | 법령 텍스트 로드 실패, 파서 전면 실패 | 중단 + 에러 리포트 |
| Error | 특정 모듈 실패, 우회 가능 | LLM API 타임아웃, MCP DB 접속 실패 | 해당 모듈 스킵 + 경고 표시 |
| Warning | 품질 저하 가능 | 오탐 보정 미적용, 조문 파싱 부분 실패 | 계속 + 리포트에 한계 기재 |
| Info | 정보 기록 | 스킵된 조문, 처리 시간 | 로그만 |

### 6.2 모듈별 Fallback

```
② 파서 실패 → 조문 수 0건 → Fatal (분석 불가)
③ 룰 패턴 1개 실패 → 해당 패턴 스킵, 나머지 계속 (Warning)
④ LLM API 전면 실패 → 역할 A: 룰 등급 유지 / 역할 B: Layer 1 템플릿만 사용 (Error)
④ LLM API 부분 실패 → 실패 건만 룰 등급 유지 (Warning)
⑤ MCP DB 실패 → L-03, L-04 스킵, 리포트에 "미검증" 표시 (Error)
⑥ 등급 산출 실패 → 발생 불가 (입력만 있으면 산술 연산)
⑦ 권고안 Layer 3 실패 → Layer 1 템플릿 사용 (Warning)
⑧ 리포트 생성 실패 → JSON 원본 제공 (Error)
```

### 6.3 리포트 내 한계 표시

에러가 발생한 경우, 리포트 Section 6(방법론)에 아래를 기재한다:

```
⚠ 분석 한계 사항:
- LLM 진단이 N건 실패하여 룰 기반 등급이 적용되었습니다
- 법령DB 연동이 불가하여 참조끊김(L-03)/위임미이행(L-04) 검증이 수행되지 않았습니다
- 조문 파싱 시 제XX조의 항/호 구조가 정확하지 않을 수 있습니다
```
